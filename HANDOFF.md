# تسليم المشروع — محبار (Mihbar)

نص بسيط لأي شخص يكمل الشغل بعدي. للتفاصيل الكاملة والتقنية شوف `MIGRATION_NOTES.md` (فيه 9 أقسام موثّقة بالكامل).

---

## ماذا فعلت (بالترتيب الزمني)

1. **رحّلت المشروع من React+Vite إلى Next.js** — إصلاح أخطاء بناء، حل مشكلة فلاش "تصميم كمبيوتر ثم هاتف" (بجعل الشل client-only عبر `next/dynamic`)، حل مشكلة نشر Vercel (`vercel.json`).

2. **راجعت كل ملفات المشروع (76 ملف)** — لقيت ولقّطت أخطاء صامتة من بقايا Vite (مسارات ميتة، service worker يخزّن ملفات مش موجودة، ألوان theme غير متطابقة).

3. **حوّلت المشروع كامل من JavaScript إلى TypeScript** (76 ملف: `.js/.jsx` → `.ts/.tsx`). اكتشفت بالطريق **5 باقات إنتاجية حقيقية** كانت مخفية (أهمها: التصويت على الاستفتاءات كان معطّل في الفيد الرئيسي، وباگ كان يرمي `TypeError` فعلي عند حذف مرفق Markdown).

4. **فحصت الأمان والأداء:**
   - لقيت ثغرة **XSS حقيقية وخطيرة** في `renderMarkdown` (اسم لغة كتلة الكود كان يُحقن بدون تعقيم) وأصلحتها.
   - صفّرت ثغرات `npm audit` (كانت 2).
   - فصلت محرر Markdown لحزمة منفصلة (code splitting) تحمّل بس وقت الحاجة.

5. **حوّلت الأنيميشن اليدوي إلى مكتبة `motion` بالكامل** (نفس Framer Motion، بس بالاسم الجديد) — 10 مكوّنات: `SettingsBottomSheet`, `FloatingPostButton`, `FloatingPostModal`, `ConfirmModal`, `DangerConfirmModal`, `DevNoticeModal`, `MdEditorPage`, `ActionMenuButton`, `MdBadge`, `PWABanner`. هذا حذف كل كتل الكود المعقّدة كانت تحاكي "دخول/خروج" العنصر يدويًا بـ `setTimeout`، ونظّف 12 `@keyframes` ميتة من `globals.css` بعدها.

**كل خطوة اتفحصت فعليًا:** بناء نظيف (`npm run build`)، اختبارات (`npm run test`، 15/15 دايمًا)، وتشغيل فعلي مع اختبار كل الروابط.

## ماذا تبقى (حسب الأولوية)

### 1. باقي انتقالات hover/press البسيطة (أولوية منخفضة، مو عاجلة)
تحويل مكوّنات "دخول/خروج" لـ `motion` **اكتمل بالكامل** (10/10: 7 من الدفعة الأولى + `ActionMenuButton.tsx`, `MdBadge.tsx`, `PWABanner.tsx` من هذه الجلسة). التفاصيل في القسم 10 من `MIGRATION_NOTES.md`.

المتبقي فقط: ~18 ملف يستخدمون `TRANSITIONS`/`ANIM` من `constants/index.ts` لانتقالات بسيطة (hover/press) — هذه ليست "دخول/خروج عنصر" بل انتقالات CSS بسيطة، تحويلها اختياري وأقل إلحاحًا.

**تم بالفعل:** نُظِّفت كل الـ `@keyframes` الميتة المرتبطة بمكوّنات "دخول/خروج" من `src/app/globals.css` (12 كتلة). الباقي في الملف (`pageEnter`, `mihbarSpin`, `threadEnter/Exit`, `toastSlideUp`, وأنيميشن التحميل المستمر) لا يزال مستخدَمًا فعليًا.

### 2. باگان بصريان موثَّقان (يحتاجوا قرار منك، مو إصلاح تلقائي)
- `PostInput.tsx`: فيه متغيّر `inputBase` (ستايل موحّد) مصمَّم يُستخدم بس مش مطبَّق فعليًا — حقل الإدخال الرئيسي شكله مختلف شوي عن حقول التعليق/الرد.
- `SHADOWS.lg` غير موجودة في `constants/index.ts` — قائمة اقتراحات الإشارة (mention dropdown) تطلع بدون ظل.

### 3. تشديد TypeScript تدريجيًا (اختياري، مستقبلي)
حاليًا `strict: false` في `tsconfig.json` — قرار مقصود عشان الترحيل الأول يكون آمن. ممكن تفعّل `strict: true` تدريجيًا (تبدأ بـ `noImplicitAny` بس) بجلسة منفصلة.

### 4. ترقية Next.js (مستقبلي، احتياط)
المشروع حاليًا على Next.js 15.5.20، وفيه إصدار 16.x متوفر. ترقية إصدار رئيسي محتاجة جلسة منفصلة مخصصة مع اختبار شامل — ما سويتها لأنها خطر على مشروع مستقر.

## إصلاحات جلسة لاحقة (بعد فحص أمان/أداء إضافي)

فُحصت ثلاث قضايا كانت موثَّقة كـ"مفتوحة" وأُصلحت كلها معًا، كل واحدة تحقّقت بشكل منفصل:

1. **تصادم معرّفات (`id`) في `usePostForm`/`useComments`/`useReplies`** — كانت تولّد `id: \`post-${Date.now()}\`` (وبالمثل reply/comment)؛ نقرة مزدوجة سريعة أو عمليتان في نفس المللي ثانية كانتا تعطيان نفس المعرّف فعليًا (استبدال عنصر بدل إضافته). الإصلاح: دالة موحّدة `generateUniqueId(prefix)` في `utils/index.ts` تضيف جزءًا عشوائيًا قصيرًا (`Math.random().toString(36)`) فوق `Date.now()`. تم التحقق فعليًا: 100,000 استدعاء متتالٍ بدون أي تصادم.

2. **`LinkPreview.tsx` بدون أنواع + طلبات متزامنة مكررة** — أُضيفت واجهات TypeScript (`LinkPreviewProps`, `LinkPreviewData`)، وأُضيف كاش وعود (`Map<string, Promise>`) على مستوى الموديول يشارك نفس طلب الـ `fetch` بين كل استدعاءات المكوّن المتزامنة لنفس الرابط (مهم لأن `LinkPreview` يظهر في `FeedPostItem`, `CommentItem`, `ReplyItem` معًا — نفس الرابط في منشور وتعليقاته كان يطلق عدة طلبات شبكة مستقلة رغم الكاش في `sessionStorage`، لأن الكاش يُكتب فقط بعد اكتمال أول طلب).

3. **`securityReady` (القضية #4 السابقة) فُعِّل فعليًا** — كان العلم موجودًا في `useMihbarSecurity.ts` لكن غير مُمرَّر لأي مكان. الآن يمر عبر `MihbarShell` → `useMihbarPosts` → `usePostForm`/`useComments`/`useReplies`، ويوقف النشر/التعليق/الرد فعليًا (`if (!securityReady || !deviceHash) return;`) خلال نافذة التحميل القصيرة عند بدء التطبيق، بدل الاعتماد فقط على فحص `deviceHash` المنفرد.

**تم التحقق:** `npm run build` نظيف (بلا أخطاء TypeScript/ESLint)، `npm run test` 15/15، وكل الروابط (`/`, `/post/[id]`, `/profile`, `/settings`) ترجع `200` بعد `npm run start` فعلي.

## قواعد مهمة لأي حد يكمل (تعلّمناها بالتجربة)

- **استيرادات TypeScript:** لازم بدون امتداد (`from "./x"` مو `from "./x.ts"`).
- **قاعدة `react-hooks/set-state-in-effect`:** أي `setState` مباشر جوا `useEffect` يوقف البناء. لو احتجت، لفّه بـ `setTimeout(..., 0)` أو الأفضل استخدم `AnimatePresence`/`onExitComplete` بدل المحاكاة اليدوية.
- **قبل أي تعديل:** دايمًا `rm -rf .next && npm run build` كامل، مو تعديل واحد وبس.
- **بعد كل دفعة تغييرات:** شغّل `npm run test` و`npm run start` واختبر الروابط فعليًا (`curl`)، مو بس البناء.
