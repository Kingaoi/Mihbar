# محبار (Mihbar) — توثيق الترحيل إلى Next.js

## نظرة عامة على المشروع

**محبار** منصة اجتماعية عربية للمشاركة المجهولة للآراء والنصائح والتجارب — بنية شبيهة بـ Reddit/Threads، RTL بالكامل، ومصممة أساسًا كتطبيق موبايل (PWA).

- **الإطار:** Next.js 15.5 (App Router) + React 19
- **التنسيق:** Tailwind CSS 4
- **قاعدة البيانات/الخدمات:** Supabase
- **الاستضافة:** Vercel
- **الحالة:** تم ترحيله من مشروع React + Vite أصلي إلى Next.js بالكامل، ويعمل حاليًا في الإنتاج.

### هيكل الصفحات (App Router)

جميع المسارات الأربعة تُركِّب نفس المكوّن الجذري (`MihbarShell` عبر `MihbarClientOnly`)، وهو تطبيق SPA-shell كامل يدير التنقل داخليًا (Home/Thread/Profile/Settings) بينما يُزامن حالته مع الـ URL:

| المسار | الغرض |
|---|---|
| `/` | الصفحة الرئيسية (الفيد) |
| `/post/[id]` | عرض منشور فردي + التعليقات (Thread) |
| `/profile` | صفحة الملف الشخصي |
| `/settings` | صفحة الإعدادات |

### الملفات المعمارية الأساسية

- `src/MihbarShell.jsx` — المكوّن الجذري؛ يحتوي كل منطق العرض والحالة المشتركة.
- `src/MihbarClientOnly.jsx` — غلاف (wrapper) يعطّل الـ SSR لـ `MihbarShell` (انظر القسم 3 أدناه لسبب هذا القرار).
- `src/hooks/` — طبقة الحالة والمنطق، مقسّمة حسب المسؤولية (`posts/`, `useBreakpoint.js`, `useMihbarConfig.js`, إلخ).
- `src/components/` — طبقة العرض، 35+ مكوّنًا منظّمًا حسب الوظيفة (`feed/`, `layout/`, `settings/`, `md_editor/`).
- `src/constants/index.js` — نظام تصميم مركزي (DRY): `CL_DARK`/`CL_LIGHT`، `TRANSITIONS`، `RADIUS`، `SHADOWS`، إلخ.

## 1. سياق الترحيل

المشروع بدأ كتطبيق **React + Vite (SPA)** واحد، ثم أُعيد هيكلته بالكامل إلى **Next.js (App Router)**. الترحيل تم من طرف المطوّر، وجاءت هذه الجلسة لتشخيص وإصلاح المشاكل الناتجة عن الانتقال بين النموذجين.

الفرق الجوهري بين النموذجين الذي تسبب في معظم المشاكل:
- **React SPA:** كل شيء يُرسم في المتصفح فقط. `window`/`document` متاحان دائمًا من أول سطر كود.
- **Next.js (SSR افتراضيًا):** كل صفحة تُرسم أولاً على السيرفر (حيث `window` غير موجود)، ثم تُعاد "ترطيبها" (hydration) في المتصفح. أي كود يفترض وجود `window` بلا حماية، أو يُنتج قيمة مختلفة بين السيرفر والعميل، يسبب أعطالًا لا تظهر في React SPA التقليدي.

## 2. إصلاح أخطاء البناء (`next build` الفاشل)

### السبب
`eslint-config-next` (v15) يفعّل قاعدة `react-hooks/set-state-in-effect` (من `eslint-plugin-react-hooks` v7)، التي تحوّل أي استدعاء **متزامن** لـ `setState` داخل جسم `useEffect` إلى **خطأ بناء قاتل** (وليس مجرد تحذير). هذا نمط كان شائعًا ومقبولًا في كود React SPA القديم (خصوصًا لمزامنة الحالة مع الـ URL)، لكنه يفشل بناء Next.js.

### الملفات المعدَّلة
- **`src/hooks/useAppUIState.js`** — حذف state + effect مكررين بالكامل؛ `settingsPageOpen`/`profilePageOpen` أصبحا **مشتقّين مباشرة** من `pathname` (عبر `usePathname()`) بدون أي `useEffect` — هذا الحل الأمثل، وليس مجرد تحايل على القاعدة.
- **`src/hooks/posts/useThreadNavigation.js`** — توحيد كل فروع منطق "مزامنة `activePostId` مع الـ URL" داخل `setTimeout(..., 0)` واحد، لضمان عدم وجود أي `setState` متزامن مباشرة داخل جسم الـ effect.
- **`src/hooks/posts/usePostForm.js`** — تأجيل `setErr("")` في عداد الـ rate-limit المتنازل بنفس الأسلوب.
- تنظيف إضافي: إزالة متغيرات `catch (e)` غير المستخدمة في 3 مواضع (`useComments.js`, `usePostForm.js`, `utils/index.js`).

### إصلاح تحذير `react-hooks/exhaustive-deps` المتبقي
الـ effect في `useThreadNavigation.js` كان مصمَّمًا عمدًا ليعمل فقط عند تغيّر `pathname` (لتفادي حلقة إعادة تشغيل ذاتية، لأن الـ effect نفسه يستدعي `setActivePostId`). الحل الصحيح: نمط **`useRef` + `useLayoutEffect`** — أحدث نسخة من كل الدوال/القيم المستخدمة تُخزَّن في `latest.current` (محدَّثة عبر `useLayoutEffect` قبل أي effect آخر)، ويقرأها الـ effect المستهدف من الـ ref بدل تمريرها كاعتمادية مباشرة. هذا يُرضي القاعدة فعليًا دون كسر السلوك المقصود أو تجاهل التحذير.

### تحسين تجميلي
إضافة `"type": "module"` في `package.json` لإسكات تحذير `MODULE_TYPELESS_PACKAGE_JSON` المتكرر في كل عملية بناء.

## 3. فلاش "تصميم كمبيوتر ثم هاتف" (المشكلة المحورية)

### الأعراض
عند فتح الموقع على الهاتف، يظهر تصميم الكمبيوتر (Desktop) للحظة، ثم يتحول فجأة لتصميم الهاتف الصحيح. هذه المشكلة **لم تكن موجودة** في نسخة React + Vite الأصلية.

### التشخيص الجذري
`useBreakpoint.js` يحسب `isMobile`/`isTablet`/`isDesktop` عبر قراءة `window.innerWidth`. بما أن `window` غير متاح على السيرفر:
1. **رندر السيرفر (SSR):** الافتراض يرجع دائمًا `"desktop"` — الـ HTML المُرسَل للمتصفح مبني بالكامل على تصميم الكمبيوتر.
2. **رندر العميل الأول (hydration):** القيمة الصحيحة (`"mobile"`) تُكتشف، لكن فقط **بعد** أول رسم فعلي على الشاشة — فيظهر الفلاش.

نفس المشكلة وُجدت أيضًا في `useMihbarConfig.js` (`systemIsDark` يعتمد على `window.matchMedia`)، وهي أوسع أثرًا لأنها تتحكم في **الألوان** (`CL_DARK`/`CL_LIGHT`) عبر كامل الشجرة.

### المحاولات والحل النهائي

**محاولة أولى (غير كافية):** حقن سكربت متزامن في `<head>` يكتب القياس الحقيقي كـ `data-bp` على `<html>` + `suppressHydrationWarning`. هذا الأسلوب هو التوصية الرسمية لـ Next.js، **لكن فقط** لعناصر بسيطة (نص واحد، أو CSS variable). في محبار، `isMobile` هو **boolean state في React** يتحكم بتفرّع شجرة JSX كاملة عبر 232 استخدامًا في 35 ملفًا (inline styles، وليس CSS classes) — فالتصحيح عبر `useEffect` بعد الـ hydration يحل خطأ الـ mismatch لكن **يُبقي الفلاش البصري**، لأن `useEffect` بطبيعته يعمل بعد أول رسم على الشاشة.

**الحل النهائي (المُطبَّق):** تعطيل الـ **Server-Side Rendering بالكامل** لمكوّن `MihbarShell`، باستخدام:

```jsx
// src/MihbarClientOnly.jsx
const MihbarShell = dynamic(() => import("./MihbarShell.jsx"), {
  ssr: false,
  loading: () => <Loader />, // شاشة تحميل بلون الخلفية الداكن (#141312)
});
```

**لماذا هذا هو القرار الصحيح لمحبار تحديدًا:**
- `MihbarShell` هو التطبيق بأكمله (نمط SPA-shell)، خلف تفاعل مستخدم بالكامل، ولا يحتاج فهرسة SEO حقيقية على مستوى الـ layout.
- بما أنه لا يوجد رندر سيرفر لهذا الجزء إطلاقًا، **لا توجد مقارنة سيرفر/عميل من الأساس** — فلا يمكن حدوث hydration mismatch بغض النظر عن القياس، ولا يظهر أي "تصميم افتراضي" مؤقت لأن لا شيء يُرسم قبل معرفة القياس الصحيح.
- البديل (إعادة هيكلة 232 استخدامًا إلى CSS variables متوافقة مع SSR) كان تغييرًا كبيرًا وخطرًا وغير مبرَّر لهذا النوع من التطبيقات.

بعد هذا القرار، أُعيدت أيضًا تبسيط `useBreakpoint.js` و`useMihbarConfig.js` إلى قراءة `window` مباشرة في القيمة الأولية (بأمان تام الآن، بما أن لا SSR لهما)، وأُزيل السكربت المؤقت من `layout.jsx` لأنه لم يعد له غرض.

**النتيجة الجانبية الإيجابية:** حجم `First Load JS` انخفض من 199 kB إلى 106 kB، لأن `MihbarShell` أصبح الآن يُحمَّل كـ chunk منفصل بعد التحميل الأولي.

**التحقق:** فحص الـ HTML المُرسَل من السيرفر أظهر `BAILOUT_TO_CLIENT_SIDE_RENDERING` — تأكيد مباشر من Next.js أن القرار طُبِّق بشكل صحيح.

## 4. مشكلة النشر على Vercel — `Output Directory "dist" not found`

### الأعراض
البناء ينجح بالكامل على Vercel (كل الصفحات تُولَّد بنجاح)، لكن يفشل النشر بخطأ:
```
Error: The Next.js output directory "dist" was not found at "/vercel/path0/dist"
```

### التشخيص
تأكدنا (عبر فحص شامل لكل الملفات) أن لا وجود لكلمة `"dist"` في أي مكان بالكود أو الإعدادات (`package.json`, `next.config.js`, إلخ). السبب كان **إعداد "Output Directory" محفوظ يدويًا في لوحة تحكم Vercel** كـ `dist` — بقايا من إعداد قديم أيام كان المشروع Vite (الذي يُخرج فعليًا إلى `dist` بشكل افتراضي)، بينما Next.js يُخرج إلى `.next`.

هذا النوع من الإعدادات (Dashboard Override) **لا يُقرأ من أي ملف في الـ repo إطلاقًا**، ولا يظهر بفحص الكود مهما دقّقنا.

### الحل
إضافة `outputDirectory` و`buildCommand` صراحة في `vercel.json`:

```json
{
  "framework": "nextjs",
  "buildCommand": "next build",
  "outputDirectory": ".next"
}
```

القيمة المحدَّدة في `vercel.json` **تتجاوز تلقائيًا** أي Override قديم محفوظ في لوحة التحكم (موثّق رسميًا من Vercel)، دون الحاجة للدخول إلى الواجهة وتعديل أي إعداد يدويًا — حل أضمن من محاولة إيجاد وإزالة الـ Override من الداشبورد، الذي أثبتت تجارب أخرى (نقاش رسمي على مجتمع Vercel) أنه قد يكون عالقًا ولا يُزال بسهولة من الواجهة.

## 6. مراجعة شاملة لكل ملفات المشروع (76 ملف)

بعد استقرار الترحيل، جرت مراجعة كاملة لكل ملف في `src/` — بحثًا عن أي أثر متبقٍ من بنية Vite القديمة (خصوصًا افتراضات حول `window`/`document` أو مسارات ملفات لم تعد موجودة)، وأي مشاكل معمارية أخرى لم تظهر بعد كأخطاء بناء أو runtime واضحة.

### نتيجة الفحص البنيوي

- **"use client" boundary:** فقط `MihbarShell.jsx` (والملفات الجذرية `MihbarClientOnly.jsx`, `ServiceWorkerRegistration.jsx`) تحمل `"use client"` صراحة. الـ 40+ مكوّنًا الأخرى في `components/` لا تحتاج التوجيه لأنها **جميعها** تُستورد فقط من تحت `MihbarShell` (مباشرة أو بالتسلسل) — تم التأكد بفحص كل استيراد في `app/*/page.jsx` أنها تستورد `MihbarClientOnly.jsx` حصرًا، دون أي استيراد مباشر لمكوّن فرعي.
- **طبقة التخزين (`dataLayer.js`, `localStorageAdapter.js`, `utils/index.js`):** موثّقة باحترافية عالية، مبنية بوعي كامل لتسهيل الانتقال المستقبلي لـ Supabase (`supabaseAdapter.stub.js` موجود كـ "عقد مطابق" غير مُفعَّل حاليًا). لا استخدام لـ `window`/`localStorage` خارج `useEffect` أو دوال مُستدعاة بفعل مستخدم — آمنة تمامًا الآن بعد تعطيل SSR لـ `MihbarShell`.
- **الاختبارات (`utils/index.test.js`):** 15 اختبارًا، كلها كانت تعمل ما عدا واحدًا.

### الأخطاء المكتشفة والمُصلَحة في هذه المراجعة

هذه أخطاء **صامتة** — لم تكن تظهر كأعطال واضحة أو تحذيرات بناء، بل كانت تُعطِّل وظائف ثانوية بهدوء بسبب فرضيات من بنية Vite لم تعد صحيحة في Next.js:

1. **`MdBadge.jsx` — `document.getElementById("root")` يُرجع دائمًا `null`.**
   Vite كان يضع React داخل `<div id="root">` (معرَّف في `index.html`)، بينما Next.js يضع React مباشرة داخل `<body>` بدون أي عنصر "root" وسيط. الكود كان يتحقق من `if (root)` قبل الاستخدام، فلم يظهر أي خطأ — لكن ميزة منع تحديد النص خلف نافذة معاينة الـ Markdown (أثناء فتحها) كانت **معطّلة بصمت بالكامل**. **الإصلاح:** استبدال `getElementById("root")` بـ `document.body` (الجذر الفعلي في Next.js).

2. **`globals.css` — قاعدة `#root` ميتة.**
   نفس السبب الجذري: `html, body, #root { ... }` — جزء `#root` من الـ selector لا يطابق أي عنصر فعليًا. `html, body` كانا لا يزالان يعملان فتُخفي المشكلة، لكن نية "امتداد الجذر لعرض كامل" لعنصر React نفسه (وليس فقط `html`/`body`) كانت مفقودة. **الإصلاح:** إزالة `#root` من الـ selector.

3. **`public/sw.js` — قائمة `ASSETS_TO_CACHE` تشير لملفات Vite غير موجودة.**
   القائمة كانت تحتوي `/index.html`, `/src/main.jsx`, `/src/App.jsx`, `/src/index.css` — لا شيء من هذا موجود في بنية Next.js (لا `index.html` منفصل، وأصول JS/CSS مبنية بأسماء hash ديناميكية تحت `/_next/static/`). محاولات التخزين هذه كانت تفشل بصمت بفضل معالجة `Promise.allSettled` الدفاعية الموجودة أصلاً في الكود (لا خطأ ظاهر في الكونسول)، لكنها كانت تُفرغ "تسخين" الكاش الأولي من أي فائدة حقيقية. **الإصلاح:** الاقتصار على الأصول ذات المسار الثابت المعروف (`/`, `/favicon.svg`, `/manifest.json`)؛ بقية أصول Next.js تُخزَّن تلقائيًا لاحقًا عبر استراتيجية Cache-First الموجودة أصلاً في الملف عند أول طلب فعلي. كما رُفعت نسخة `CACHE_NAME` (v1 → v2) لضمان مسح كاش أي مستخدم قديم يحمل الأصول الميتة.

4. **`public/manifest.json` — لون ثيم غير متزامن (`#1C1917` بدل `#141312`).**
   `theme_color`/`background_color` في المانيفست كانا يستخدمان لونًا مختلفًا عن `CL_DARK.bg` الفعلي (`#141312`) المستخدم في باقي التطبيق (`layout.jsx`, شاشة التحميل). هذا يُنتج تضاربًا بصريًا ملموسًا (لون status bar/splash screen مختلف عن خلفية التطبيق الفعلية) عند تثبيت الـ PWA. **الإصلاح:** توحيد القيمتين مع `#141312`.

5. **اختبار قديم فاشل في `utils/index.test.js`** (غير متعلق بالترحيل لـ Next.js مباشرة).
   اختبار `"should flush code blocks even if not closed"` كان يتحقق من وجود نص خام (`"const x = 5;"`) داخل ناتج `renderMarkdown`، لكن الدالة الآن تُطبِّق تلوين أكواد عبر Prism (تحويل النص إلى عناصر `<span class="token...">`) — الاختبار لم يُحدَّث بعد إضافة هذه الميزة. **الإصلاح:** تغيير لغة الكود التجريبية في الاختبار إلى `plaintext` (غير مسجَّلة في Prism)، بحيث يبقى الاختبار مركّزًا على السلوك المقصود أصلاً (الإغلاق التلقائي لكتلة كود غير مُغلَقة) دون الاقتران بتفاصيل تنفيذ التلوين.

6. **`metadataBase` غير معرَّف** (كان مذكورًا كتحسين مؤجَّل في القسم السابق) — أُضيف الآن فعليًا في `layout.jsx`.

### دَين تقني موثَّق (بدون تعديل — قرارات تصميم مؤجَّلة، وليست أخطاء ترحيل)

- **`useMihbarSecurity.js` — `securityReady` مُصدَّر لكن غير مُستهلَك.** الـ hook يحسب علمًا (`securityReady`) يصبح `true` فقط بعد اكتمال تحميل `deviceHash`/حالة الحظر من التخزين، والتعليق البرمجي يوضّح أن أي إجراء حساس (نشر منشور) يجب أن ينتظر هذا العلم. لكن `MihbarShell.jsx` لا يستخرج هذا العلم من `security` حاليًا، ولا يُستخدم في أي مكان بالمشروع. `usePostForm.js`/`useComments.js`/`useReplies.js` لديها فحص دفاعي بديل (`if (!deviceHash) return;`) يُغطي جزءًا من نفس المخاطرة عمليًا. هذا يبدو استعدادًا مقصودًا لتفعيل لاحق أكثر منه نقصًا حرجًا — لم يُعدَّل لأنه قد يكون قرار تصميم مؤجَّل عمدًا من المطوّر.
- **`useDocumentMetadata.js` — تعديل meta tags من جهة العميل بعد التحميل.** موثَّق بوضوح في الملف نفسه أن الحل الأصح هندسيًا (بعد الربط بـ Supabase) هو تحويل `app/post/[id]/page.jsx` لـ Server Component يستخدم `generateMetadata`، لكن هذا غير ممكن حاليًا لأن كل البيانات في `localStorage` (لا يوجد مصدر بيانات يقدر عليه الخادم وقت الطلب).

## 7. الترحيل من JavaScript إلى TypeScript

بعد استقرار مشروع Next.js بالكامل، جرى ترحيل شامل لكل الـ 76 ملفًا في `src/` من JavaScript/JSX إلى TypeScript/TSX، على 4 مراحل متتالية مع تحقق كامل (بناء + اختبارات + تشغيل فعلي) بعد كل مرحلة.

### الإعداد
- `strict: false` مؤقتًا في `tsconfig.json` — قرار مدروس لمشروع بهذا الحجم (76 ملفًا، منطق أعمال معقّد). الهدف الأول: ترحيل آمن (compile ناجح، صفر تغيير سلوكي)، وليس تشديد الأنواع الفوري. تشديد `strict: true` تدريجيًا خطوة منفصلة مستقبلية.
- `next-env.d.ts` تولّد تلقائيًا عبر `next build` (لا يُعدَّل يدويًا).
- `src/globals.d.ts` أُضيف يدويًا لإعلان وحدات CSS (`declare module "*.css"`) — مطلوب لأن `import "./globals.css"` (side-effect import) في `layout.tsx` لا يُتعرَّف عليه من TypeScript افتراضيًا.

### مراحل التنفيذ
1. **`hooks/`** (16 ملف) — الأساس المنطقي، لا JSX.
2. **الطبقات المساعدة** (`utils/`, `constants/`, `data/`, `locales/`, `styles/`, 11 ملف) — الأوسع أثرًا على باقي المشروع (50 ملفًا آخر احتاجت تحديث استيراداتها).
3. **`components/`** (38 ملف `.tsx`) — الأكبر والأكثر تعقيدًا.
4. **`app/` + الملفات الجذرية** (`MihbarShell.tsx`, `MihbarClientOnly.tsx`, 8 ملفات) — المرحلة الأخيرة.

قاعدة تقنية أساسية طُبِّقت باستمرار: في TypeScript، استيرادات الملفات المحلية **يجب ألا تحمل امتداد صريح** (`from "./x"` وليس `from "./x.ts"` أو `from "./x.tsx"`) — بخلاف JavaScript ESM القياسي الذي كان يتطلب الامتداد صراحة (`from "./x.js"`).

### باگات إنتاجية حقيقية اكتُشفت وأُصلحت بفضل TypeScript

هذه أخطاء **لم تكن لتظهر أبدًا في JavaScript العادي** — TypeScript يفحص تطابق كل `prop` بين المُرسِل والمُستقبِل، بينما JavaScript يتجاهل بصمت أي خاصية زائدة أو مفقودة عند الـ destructuring:

| # | الموقع | الباگ | الأثر الفعلي |
|---|---|---|---|
| 1 | `FeedItem.tsx` | لا يستقبل `handlePollVote` ولا يُعيد تمريرها لـ `FeedPostItem` | **التصويت على الاستفتاءات كان معطّلاً في الفيد الرئيسي** (يعمل فقط داخل صفحة الـ Thread، حيث يُمرَّر مباشرة) |
| 2 | `PostInput.tsx` | `setMdPreview(false)` يُستدعى على دالة **غير موجودة إطلاقًا** في أي مكان بالمشروع (لا state، لا تعريف) | **`TypeError` فعلي في المتصفح** عند إزالة مرفق Markdown من منشور جديد |
| 3 | `useMihbarPosts.ts` | `savedPosts` يُمرَّر لـ 3 hooks (`useComments`, `useReplies`, `usePostEditing`) لا تستقبلها ولا تستخدمها | تمرير بيانات ميتة، صفر أثر وظيفي لكنه تلوّث معماري |

**كل هذه أُصلحت بإزالة التمرير الميت** (باستثناء #1 و#2 اللتين احتاجتا إصلاحًا فعليًا لاستعادة وظيفة كانت معطّلة/مكسورة).

### باگات تصميمية موثَّقة (لم تُعدَّل عمدًا — قرار يخص المطوّر)

- **`PostInput.tsx` / `inputBase`**: مصمَّم للاستخدام (`...inputBase` كستايل موحّد)، ومُستخدَم فعليًا في كل مكوّنات الإدخال الأخرى (`CommentItem`, `ReplyItem`, `ThreadView`, `MdEditorPage`) — **إلا** `PostInput` نفسه، حيث لا يُطبَّق إطلاقًا. احتمال تناسق بصري غير مكتمل لحقل الإدخال الرئيسي.

كلا الباگين أُبقيا على سلوكهما الأصلي بالضبط (نفس النتيجة البصرية القديمة)، مع توثيق تفصيلي في مكان الكود نفسه وهنا، بدل افتراض النية الصحيحة وتغيير المظهر بدون تفويض صريح.

### إصلاحات نوع تقنية بحتة (صفر تأثير سلوكي)

أنماط متكررة عولجت بشكل موحَّد عبر المشروع فور اكتشافها أول مرة (بحث استباقي شامل بدل انتظار كل خطأ فرديًا):

## 8. فحص الأمان والأداء

فحص مستقل بعد اكتمال ترحيل TypeScript، بحثًا عن ثغرات أمنية وفرص تحسين أداء حقيقية — كل نتيجة تحققت منها فعليًا (تشغيل اختبارات اختراق فعلية، قياس أحجام حزم فعلية)، وليس تخمينًا نظريًا.

### 🚨 ثغرة أمنية حرجة مُصلَحة: Stored XSS عبر `renderMarkdown`

**الاكتشاف:** `codeLang` (اسم اللغة المُستخرَج من كتلة كود Markdown، مثل ` ```javascript `) كان يُحقَن مباشرة في `class="language-${codeLang}"` **بدون أي تعقيم**، بينما محتوى الكتلة نفسه كان مُعقَّمًا بشكل صحيح عبر `escapeHtml`. تم إثبات الثغرة فعليًا (وليس نظريًا) بتشغيل `renderMarkdown` على payload حقيقي:

```
```"><img src=x onerror=alert(1)>
some code
```
```

النتيجة: HTML يحتوي `<img src=x onerror=alert(1)>` قابل للتنفيذ فعليًا. بما أن `renderMarkdown` تُستخدَم عبر `dangerouslySetInnerHTML` لعرض محتوى **مُدخَل من مستخدمين آخرين** (منشورات، تعليقات تحتوي مرفقات Markdown)، هذه ثغرة **Stored XSS كاملة** — أي مستخدم يقدر يكتب منشورًا بهذا الـ payload، وأي زائر آخر يفتحه يُنفَّذ الكود في متصفحه.

**الإصلاح:** تقييد `codeLang` لأحرف/أرقام/شرطات فقط (`.replace(/[^a-z0-9-]/g, "")`) عند نقطة الاستخراج — الصيغة الوحيدة المنطقية لاسم لغة برمجة أصلاً، فلا تأثير على أي استخدام مشروع. تم التحقق بإعادة تشغيل نفس الـ payload بعد الإصلاح (نتيجة آمنة)، والتأكد من استمرار عمل تلوين الأكواد الطبيعي (`javascript`, `python`, إلخ) بلا تغيير. اختبار الاختراق أُضيف ضمنيًا عبر `utils/index.test.ts` (15 اختبارًا، منها اختبارات `renderMarkdown` الموجودة أصلاً التي استمرت بالنجاح).

**نقاط أخرى فُحصت واعتُبرت آمنة فعليًا (تم اختبارها، وليس افتراضها):**
- `renderInlineMd` (روابط وصور Markdown `[نص](url)`): محمية بازدواجية — `escapeHtml` يُطبَّق على السطر بأكمله قبل أي معالجة، والـ regex يقبل فقط `http(s)://`، فلا مجال لـ `javascript:` أو كسر الـ attribute حتى بمحاولات مباشرة (تم اختبار 4 payloads مختلفة فعليًا).
- روابط الفيديو الخارجية (`VideoPlayer.tsx`, `LinkPreview.tsx`): معرّفات الفيديو (Vimeo/YouTube) مُستخرَجة عبر regex يضمن أرقامًا فقط؛ روابط `LinkPreview` تمر عبر `encodeURIComponent` بشكل صحيح.

### ⚠️ قيد معماري موثَّق (غير قابل للإصلاح بتعديل كود — قرار بنيوي)

نظام الحظر (`checkIfBanned`/`banDevice`) وحدود معدل الاستخدام (`canPerformAction`) يعملان بالكامل من طرف العميل عبر `localStorage`، مع آلية "توقيع/checksum" (`verifyGlobalIntegrity`) تكتشف تعديل البيانات المباشر. هذه الآلية **رادعة بسيطة فقط** — أي مهاجم قادر تقنيًا على قراءة منطق حساب التوقيع (مكشوف بالكامل في كود JavaScript المُرسَل للمتصفح) وحساب توقيع صحيح لبيانات معدَّلة. هذا **ليس باگًا** بل نتيجة طبيعية لتصميم "localStorage كبنية تحتية مؤقتة" الموثَّق مسبقًا (القسم 1) بانتظار الربط بـ Supabase. الحل الحقيقي الوحيد: تحقق من صلاحيات من طرف خادم فعلي (Supabase RLS بدل منطق العميل).

### 🔒 ثغرة تبعية: `postcss` (متوسطة الخطورة، CVE مُؤكَّد)

`npm audit` كشف ثغرة XSS معروفة (`GHSA-qx2v-qp2m-jg93`) في `postcss@8.4.31` — نسخة **داخلية مُجمَّدة** ضمن `next@15.5.20` نفسها (وليست تبعية مباشرة نتحكم بها). التبعيات المباشرة للمشروع (`@tailwindcss/postcss`, `autoprefixer`, `vitest`) كانت أصلًا على النسخة الآمنة (`8.5.16`).

**الإصلاح:** إضافة `"overrides": { "postcss": "^8.5.10" }` في `package.json` — يفرض النسخة الآمنة على كل التبعيات المتداخلة بما فيها الداخلية لـ Next.js، **دون** ترقية Next.js نفسها (البديل المقترح تلقائيًا عبر `npm audit fix --force` كان سيُنزّل Next.js إلى إصدار `9.x` القديم جدًا — خيار مرفوض). تم التحقق: `npm audit` يُظهر الآن **صفر ثغرات**، والبناء يعمل بلا أي مشكلة توافق.

*ملاحظة منفصلة: Next.js 16.2.10 متاحة حاليًا (نحن على 15.5.20). ترقية إصدار رئيسي كبيرة الخطورة على مشروع مستقر — لم تُطبَّق ضمن هذا الفحص، وتحتاج جلسة منفصلة مخصصة مع اختبار شامل إن رغب المطوّر بها لاحقًا.*

### ⚡ تحسين أداء: Code-splitting لمحرر Markdown

**الاكتشاف:** `MdEditorPage` (ومعه `Prism` وكل لغاته السبع المستوردة) كان يُستورَد بشكل **ثابت (static)** في `MihbarShell.tsx`، رغم أنه مُرسَم شرطيًا فقط (`{mdEditorState && (<MdEditorPage .../>)}`) — يعني هذا الكود يُحمَّل ضمن الحزمة الرئيسية لكل مستخدم من أول لحظة تصفح، حتى لو لم يفتح المحرر إطلاقًا.

**الإصلاح:** تحويل الاستيراد إلى `next/dynamic` مع شاشة تحميل (`Loader`) أثناء تحميل الحزمة الفعلي عند أول فتح للمحرر. تم التحقق فعليًا بمقارنة أحجام الـ chunks المبنية قبل/بعد: الـ chunk الرئيسي المدمج (180K) انقسم إلى chunk رئيسي أصغر (172K) + chunk منفصل لمحرر Markdown (8K) يُحمَّل عند الطلب فقط.

**قرار بالمقابل — لم يُطبَّق نفس النمط على `MdBadge.tsx`:** رغم أنه يستورد نفس `renderMarkdown`/`Prism`، يظهر **مباشرة ضمن الفيد الرئيسي** (`FeedPostItem`, `CommentItem`, إلخ) وليس خلف تفاعل مستخدم لاحق — تحويله لـ dynamic كان سيُسبب وميض تحميل ملحوظ لكل مستخدم يتصفح منشورًا يحتوي مرفقًا، مقابل توفير هامشي (Prism أصلًا خفيفة الحجم). القرار الصحيح هنا: الإبقاء عليه ثابتًا.

### فرص أداء أخرى فُحصت واعتُبرت غير قابلة للتحسين حاليًا (أو غير ضرورية)

- **`FeedPostItem` بلا `React.memo`:** غير ضروري — الأب `FeedItem` محمي بـ `memo` بالفعل، فيحمي فعليًا من إعادة الرسم المتكررة على مستوى الشجرة الصحيح.
- **`next/image`:** غير قابل للتطبيق مباشرة — لا يوجد `<img>` كـ JSX tag في المشروع؛ كل الصور مبنية كـ HTML خام داخل `dangerouslySetInnerHTML` عبر `renderInlineMd` (نتيجة تصميم Markdown renderer مخصص). تفعيله يتطلب إعادة هيكلة الـ renderer لإرجاع React elements بدل نص HTML — تغيير معماري كبير خارج نطاق هذا الفحص.
- **تحميل المنشورات دفعة واحدة (`usePostsData`):** مقبول لحجم مشروع بهذا الحجم يعتمد على `localStorage` بدون صفحات server-side حقيقية؛ `useMemo` مُستخدَمة بالفعل لمنع إعادة حساب الفلترة غير الضرورية.

## 9. الانتقال التدريجي إلى Motion (Framer Motion)

بدأ انتقال تدريجي (وليس شاملاً) من نظام الأنيميشن اليدوي (`@keyframes` CSS + `useState`/`useEffect`/`setTimeout` لإدارة دخول/خروج العناصر) إلى مكتبة `motion` (الاسم الرسمي الجديد لـ Framer Motion منذ 2025 — نفس الفريق، نفس الـ API، `import from "motion/react"`)، المُثبَّتة أصلًا في المشروع (`motion@^12.42.2`) ومُستخدَمة سابقًا فقط في `MainLayout.tsx`.

### الدافع

النمط اليدوي القديم لإدارة دخول/خروج العناصر (state إضافي مثل `isClosing` + `useEffect` + `setTimeout` مطابق لمدة الأنيميشن) كان المصدر المباشر لـ **3 من أخطاء `react-hooks/set-state-in-effect`** التي واجهناها أثناء الترحيل لـ Next.js (`SettingsBottomSheet.tsx`, `MihbarShell.tsx` ×2). `AnimatePresence` من Motion يحل هذا بنيويًا: يعرف تلقائيًا متى يجب تشغيل أنيميشن الخروج ومتى يُزال العنصر فعليًا من DOM، دون أي state/timeout يدوي.

### الملفات المُحوَّلة (الدفعة الأولى)

| الملف | التغيير |
|---|---|
| `SettingsBottomSheet.tsx` | إزالة `isOpenLocal`/`isClosing`/`useEffect` المعقّد (كان يحتوي `setTimeout` مزدوجًا) بالكامل → `AnimatePresence` + `motion.div` |
| `FloatingPostButton.tsx` | إزالة `floatingBtnVisible`/`floatingBtnClosing` كـ props مُدارَة من الأب → يستقبل `isVisible` مباشرة، يدير الأنيميشن داخليًا |
| `FloatingPostModal.tsx` | نفس النمط: `floatingModalVisible`/`floatingModalClosing` → `isOpen` مباشرة |
| `MihbarShell.tsx` | حذف `floatingBtnVisible/Closing`, `floatingModalVisible/Closing`, و**effectين كاملين** (كل واحد فيهما بنمط `setTimeout` مزدوج) بعد نقل المنطق للمكوّنات الفرعية |
| `ConfirmDialogs.tsx` (`ConfirmModal`, `DangerConfirmModal`) | لم يكن فيهما `setTimeout` يدوي أصلاً (تختفي فجأة بدون exit animation) — أُضيف أنيميشن خروج ناعم لم يكن موجودًا، كتحسين UX إضافي بجانب توحيد النمط |
| `DevNoticeModal.tsx` | استبدال `isClosing` + `setTimeout(250)` (تخمين يدوي لمدة الأنيميشن) بـ `onExitComplete` — callback رسمي من `AnimatePresence` يُستدعى بعد انتهاء أنيميشن الخروج **الفعلي**، أدق من أي توقيت مخمَّن |
| `MdEditorPage.tsx` + `MihbarShell.tsx` | نقل إدارة الدخول/الخروج من الأنيميشن الثابت (`mdEditorFadeIn`) في المكوّن نفسه إلى `AnimatePresence` يلف نقطة الاستدعاء في `MihbarShell.tsx` — متوافق مع `next/dynamic` (المُطبَّق مسبقًا على هذا المكوّن) لأن `AnimatePresence` يتتبع فقط وجود/غياب العنصر في الشجرة، بغض النظر عن آلية تحميله الداخلية |

### ما لم يُحوَّل بعد (متبقٍ لدفعات لاحقة إن رغب المطوّر)

- `ActionMenuButton.tsx`, `MdBadge.tsx`, `PWABanner.tsx` — لا تزال تستخدم `fadeInOverlay`/`popInCard` من `globals.css`.
- ~18 ملفًا إضافيًا يعتمدون على `TRANSITIONS`/`ANIM` الثابتة العامة (`colorChange`, `press`, إلخ) لانتقالات لونية/تفاعلية بسيطة (hover/press) — هذه أبسط من مشكلة "دخول/خروج" وأقل إلحاحًا للتحويل، لكن نفس المبدأ ينطبق.
- الـ`@keyframes` غير المستخدَمة بعد التحويل (`fadeOutOverlay`, `slideUpSheet`, `slideDownSheet`, `popOutCard`, `scaleUpNotice`, `scaleDownNotice`, `fabEnter`, `fabExit`, `mdEditorFadeIn`) **لم تُحذَف بعد** من `globals.css` عمدًا — بعضها (`fadeInOverlay`, `popInCard`) لا يزال مُستخدَمًا من الملفات الثلاثة المتبقية أعلاه؛ التنظيف الكامل يُفضَّل بعد اكتمال التحويل الشامل.

### التحقق

بناء نظيف كامل (صفر أخطاء/تحذيرات) + 15/15 اختبار + 4/4 مسار بحالة `200`، بعد كل الدفعات مجتمعة.

## 10. إكمال تحويل Motion (الدفعة الثانية والأخيرة) + تنظيف الـ CSS

استُكمل تحويل الأنيميشن اليدوي للمكوّنات الثلاثة المتبقية من القسم 9، ثم حُذفت الـ `@keyframes` الميتة من `globals.css`.

### الملفات المُحوَّلة

| الملف | التغيير |
|---|---|
| `ActionMenuButton.tsx` | القائمة المنسدلة (dropdown) كانت تُخفى فجأة (`{isOpen && <div animation="popInCard ANIM.fast">}`) بدون exit animation. حُوِّلت إلى `AnimatePresence` + `motion.div` (`scale 0.96→1`, `0.12s`, نفس توقيت `ANIM.fast` الأصلي) |
| `MdBadge.tsx` | مكوّن مزدوج الطبقة (overlay + card) داخل `createPortal`. كان الـ portal نفسه مشروطًا بـ `isOpen` (`{isOpen && createPortal(...)}`) مما يمنع أي exit animation فعليًا. أُعيد الهيكلة: الـ portal الآن دائم التركيب، و`AnimatePresence` هو من يتحكم بالعرض/الإخفاء داخله — overlay يتلاشى (`fadeInOverlay` → `opacity` transition)، والبطاقة تكبر (`popInCard` → `scale 0.96→1`, `0.3s`, نفس منحنى `ANIM.sheet`) |
| `PWABanner.tsx` | كان يستخدم `if (!showPwaBanner \|\| !deferredPrompt) return null` — يختفي فورًا بلا انتقال. حُوِّل الشرط إلى متغيّر `isVisible` يُغذّي `AnimatePresence`، فصار البانر يدخل/يخرج بانتقال ناعم (`scale 0.96→1`, `0.2s`) بدل الاختفاء المفاجئ |

كل الثلاثة يتبعون نفس نمط الدفعة الأولى: `import { motion, AnimatePresence } from "motion/react"`, بدون أي state أو `setTimeout` يدوي إضافي، وبنفس قيم التوقيت/المنحنى (easing) التي كانت في الـ `@keyframes` الأصلية — أي **لا تغيير بصري** متعمَّد، فقط استبدال الآلية.

### تنظيف `globals.css`

بعد التأكد (بحث شامل في `src/**/*.{ts,tsx,css}`) أن لا شيء يستخدم الأسماء التالية بعد الآن، حُذفت 12 `@keyframes` ميتة بالكامل:

`mdEditorSlideIn`, `mdEditorFadeIn`, `slideUpSheet`, `slideDownSheet`, `fadeInOverlay`, `fadeOutOverlay`, `popInCard`, `popOutCard`, `fabEnter`, `fabExit`, `scaleUpNotice`, `scaleDownNotice`

**لم تُمس** (لا تزال مستخدَمة فعليًا من مكوّنات لم تُحوَّل بعد ضمن قائمة الـ~18 ملف المتبقية في `HANDOFF.md`، خارج نطاق هذه الجلسة):
- `pageEnter` — `SettingsPage.tsx`, `ProfilePage.tsx`
- `mihbarSpin`, `threadEnter`, `threadExit` — `ThreadView.tsx` وأماكن أخرى
- `toastSlideUp` — `Toast.tsx`
- `mihbar-shine`, `mihbar-float-up`, `mihbar-premium-spin`, `mihbar-pulse` — أنيميشن مستمر (loading/reactions)، خارج نطاق تحويل "دخول/خروج"

### الحالة بعد هذه الجلسة

- **10/10 مكوّنات "دخول/خروج"** محوَّلة بالكامل إلى `motion` (7 من الدفعة الأولى + 3 من هذه الجلسة). لا يوجد أي مكوّن يستخدم النمط اليدوي (`isClosing` + `setTimeout`) أو CSS `animation:` لمحاكاة دخول/خروج بعد الآن.
- المتبقي من القسم 9 (~18 ملفًا لانتقالات `TRANSITIONS`/`ANIM` البسيطة كـ hover/press) **لم يُمس عمدًا** — هذه ليست "دخول/خروج عنصر" بل انتقالات لونية بسيطة، أولويتها منخفضة كما هو موثَّق في `HANDOFF.md`.

### التحقق

- `rm -rf .next && npm run build` → نظيف بدون أخطاء/تحذيرات (بما فيها فحص الأنواع/الـ lint).
- `npm run test` → 15/15 (لا تغيير، كما هو متوقَّع — التحويل بصري بحت).
- `npm audit` → 0 ثغرات (بلا تغيير).
- بحث شامل بعد الحذف تأكيدًا أن لا مرجع باقٍ لأي من الـ12 `@keyframes` المحذوفة في أي ملف `.ts`/`.tsx`/`.css`.
